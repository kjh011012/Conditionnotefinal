[FIGMA MAKE UPGRADE PROMPT v3 — 컨디션노트 SaaS (멀티 치유마을 · 기간 종속 · 역할 기반 · 동적 모드 전환)]

당신은 “컨디션노트(Condition Note)”의 수석 프로덕트 디자이너이며,
본 프로젝트는 단일 앱이 아니라 “100개+ 치유마을이 참여하는 멀티 테넌트 운영 SaaS 플랫폼”입니다.

현재 상태:
- 참가자(40대 이상 성인)용 기본 컨디션/건강/리듬 화면(일반 기능)은 이미 충분히 구축되어 있습니다.
- 그러나 지금 UI는 “한 마을 참여 기반”의 느낌이 약하고, 여러 마을 데이터가 섞여 보이며,
  참가자/보호자/코디네이터/관리자(가맹주) 권한 분리, 기간 종속 참여 모드 전환, 결제→참여 생성 흐름,
  마을별 데이터 완전 분리, 마을별 운영(프로그램/식단/숙소/공지/문의/댓글) 기능이 부족합니다.

이번 작업 목표(절대 조건):
1) 멀티 테넌트(치유마을) 구조를 UI/권한/데이터 스코프에서 “명확히 체감”되게 설계.
2) 사용자의 “기간 종속 참여(Participation)”에 따라 NORMAL_MODE ↔ VILLAGE_MODE가 자동 전환되는 구조를 UI에 자연스럽게 반영.
3) 역할(참가자/보호자/코디네이터/관리자)의 기능은 “현재 마을 컨텍스트(village context) + 참여 기간”에 따라 동적으로 노출.
4) 관리자(가맹주)는 ‘자기 마을 1개’만 관리하며, 다른 마을 전환 UI가 존재하면 안 됨(테넌트 고정).
5) 코디네이터는 캠프(프로그램) 운영과 체크리포트 중심의 “운영 OS”가 되어야 하며, 참가자 화면 복제 금지.
6) 캠프 찾기/소개/추천(탐색)은 “운영 모드와 분리”된 모듈로 유지(일반 모드 중심).
7) 기존 참가자 일반 기능(건강 기록/분석/리포트 등)은 그대로 유지(삭제/대체 금지). 단, 아동/놀이 톤은 성인(40+) 톤으로 교정.

필수 제약:
- iOS/Android 완전 동일 UI(레이아웃 100% 동일).
- 모든 문구/라벨/카피는 한국어.
- 의료 진단/확정/치료 표현 금지. “관리/지원/권고/참고/가능성” 톤.
- 측정기/워치 데이터는 “측정→서버 저장→앱 조회”이며 직접 연동 조작 UI가 아니라 “동기화 상태/마지막 수집” 중심.
- 개인정보/공유는 성인 대상 원칙: 동의 기반(Consent) + 기간 기반(Participation) 스코프 제한.

0) 핵심 개념 정의 (반드시 UI에 반영)
========================================================
[개체(Entity)]
- Village(치유마을): 100+개, 각 마을은 고유 브랜드(로고/컬러/소개)를 가짐.
- Program/Camp(프로그램/캠프): 각 마을 내부에서 운영되는 기간형 상품(예약/결제 대상).
- Participation(참여): user_id + village_id + program_id + start_date + end_date
  - 기간(start_date~end_date) 동안만 “마을 참가 모드(VILLAGE_MODE)”가 활성화됨.
- Reservation/Payment(예약/결제): 결제 완료 시 자동으로 Participation 생성(시작 전 대기 가능).
- Role(역할): Participant(참가자), Guardian(보호자/자녀), Coordinator(코디네이터), Admin(가맹주/마을 관리자), (옵션) PlatformAdmin(본사 슈퍼관리자).

[모드(Mode)]
- NORMAL_MODE (일반 모드): 현재 활성 참여가 없음.
  - 기본 대시보드(개인 건강/리듬/인지/정서 루틴) 표시
  - 탐색/예약/결제 가능
  - 과거 참여 기록 조회 가능
- VILLAGE_MODE (마을 참가 모드): 활성 참여가 있음.
  - 참여 기간 동안 UI가 해당 마을/프로그램 컨텍스트에 종속(브랜드 적용 + 전용 기능 활성화)
  - 화면 상단에 “현재 참여 마을 + 남은 일정(남은 일수)” 표시
  - 마을 관계 기반 권한 UI 적용
  - 기간 종료 시 자동으로 NORMAL_MODE로 부드럽게 전환

[자동 분기 로직(UX에 자연스럽게 녹여야 함)]
if user.hasActiveParticipation():
    mode = VILLAGE_MODE
else:
    mode = NORMAL_MODE

추가: 여러 참여가 동시에 겹칠 수 있으므로(확장 대비)
- 활성 참여가 2개 이상이면: “현재 참여 공간 선택(입장)” 화면을 먼저 보여준다.
- 과거/미래 예약 참여는 “일정(타임라인)”에서 확인 가능.

========================================================
1) 디자인 요구사항 (감성/브랜딩/혼란 방지)
========================================================
- “종속” 느낌 금지. 대신 “참여하는 공간에 들어간다(입장)” 감성적 전환.
- VILLAGE_MODE 진입 시:
  - 풀스크린 “입장(Enter Village)” 트랜지션 화면(마을 로고/브랜드 컬러/환영 문구)
  - ‘오늘 일정 하이라이트’ 1줄 표시(불안감 없이 기대감)
- 마을별 브랜딩:
  - 로고/대표 컬러/보조 컬러를 토큰화(theme tokens)하여 헤더, 배지, 강조 버튼에만 적용
  - 기본 가독성/신뢰감은 유지(과한 색 면적 금지)
- 컨텍스트 표시(혼란 방지):
  - VILLAGE_MODE의 모든 화면 상단에 “컨텍스트 바(Context Bar)”를 고정 표시:
    - “마을명 · 프로그램명 · 기간” + 남은 일수(D-)
    - 예: “고라데이 치유마을 · 3월 치유캠프(03.10~03.17) · D-3”
    - ‘전환’ 버튼(참가자/보호자는 내 참여 중/예정 목록, 코디/관리자는 배정된 것만)
- 참여 종료:
  - “수료/완료” 화면(요약 리포트 + 다음 안내) 후 NORMAL_MODE로 부드러운 전환 애니메이션

톤:
- 전문적, 차분, 의료 데이터 기반 신뢰감
- 40대 이상 대상(성인) 문구
- 접근성: 큰 글자 모드 고려, 터치 영역 44px 이상, 색상+텍스트 병행

========================================================
2) 정보 구조(IA) — 모드별 2층 구조
========================================================
[1층: NORMAL_MODE(글로벌)]
- 홈(개인 대시보드: 기존 유지)
- 캠프/프로그램 찾기(소개/추천/필터/상세)
- 내 예약/결제(예약 상태/결제/참여 확정)
- 내 참여 이력(과거/현재/미래 일정 타임라인)
- 가족/연결(보호자 연결/관리)
- 설정

[2층: VILLAGE_MODE(마을 컨텍스트 내부)]
- (공통) 컨텍스트 바 고정
- 마을 전용 대시보드(진행 바/일정/코디/공지/문의/댓글/연결상태)
- 프로그램 일정(타임라인/세션)
- 식단(표시/공지)
- 숙소(참가자·보호자: 조회 중심 / 운영자: 배정·관리)
- 문의/댓글/공지(마을 전용)
- 역할별 운영 기능:
  - 코디네이터: 캠프보드/체크리포트/세션진행/이슈
  - 관리자: 프로그램/식단/숙소 CRUD, 공지/문의/댓글 관리, 예약/결제 확인, 통계

“캠프 찾기”는 NORMAL_MODE 중심이며, VILLAGE_MODE에서는 메인 탭/네비에 노출하지 않고 ‘더보기’ 정도로 최소화.

3) 기간 종속 참여(Time-Scoped Participation) — UI가 반드시 지원해야 할 것
========================================================
- 과거 참여 이력 리스트(마을/프로그램/기간/완료 상태)
- 현재 참여 마을 강조 표시(활성 참여)
- 향후 예약 마을 일정 표시(시작 전 대기)
- 마을별 데이터 완전 분리:
  - 공지/문의/댓글/리포트/체크리포트/숙소/식단/프로그램 일정은 village_id + program_id 스코프로만 표시
  - 다른 마을 데이터가 섞여 보이면 실패

========================================================
4) 역할 관계 구조(Village Context Role System) — 기간 동안만 동적 관계 형성
========================================================
- 참가자 ↔ 보호자:
  - 보호자는 “해당 마을 참여 기간 동안만” 접근 가능
  - 기간 종료 시 접근 차단(또는 종료 안내 화면)
- 참가자 ↔ 코디네이터:
  - 코디네이터는 해당 프로그램 진행/상태 업데이트/체크리포트 관리
- 참가자 ↔ 관리자(가맹주):
  - 마을 운영 데이터 종합 관리, 공지, 문의/댓글 모더레이션
- 보호자 ↔ 코디네이터:
  - 참가자 상태 공유(동의+기간 기반)
- 보호자 ↔ 관리자:
  - 보호자 문의 및 요청 처리
- 코디네이터 ↔ 관리자:
  - 진행상황 공유/리포트/운영 데이터 관리(Closed Loop)

UI 설계 시, “현재 마을 컨텍스트 + 참여 기간 + 사용자 역할”에 따라 노출 기능이 달라져야 함.

========================================================
5) 결제 → 참여 생성 흐름(Reservation → Payment → Participation)
========================================================
- 예약/결제 완료 시: Participation 자동 생성(start_date~end_date 포함)
- 시작 전 대기 상태 UI:
  - “참여 확정” + “시작까지 D-n”
  - 사전 안내/준비물/공지 보기 가능(옵션)
- 참여 기간 동안: VILLAGE_MODE 자동 활성화
- 기간 종료 시: 참여 종료 + 완료 처리 + 수료 화면 → NORMAL_MODE 복귀

UI에 반드시 포함:
- 예약 상태 표시(임시예약/결제대기/결제완료/참여확정/진행중/완료/취소/환불)
- 결제 완료 → 참여 확정 표시
- 시작 전 대기 상태 화면
- 종료 후 완료 표시(요약 리포트 제공)

========================================================
6) 관리자(가맹주) 접근 제한(멀티 테넌트 SaaS 핵심)
========================================================
- 가맹주는 자신이 운영하는 마을 데이터만 관리 가능
- 관리자 UI에 ‘마을 전환’이 보이면 안 됨(테넌트 고정 헤더)
- (옵션) PlatformAdmin 백오피스만 마을 생성/가맹주 계정 생성/할당 가능

7) 공통 UI 컴포넌트/시각화(필수)
========================================================
- Context Bar(마을명·프로그램명·기간·남은일수 + 전환)
- Village Theme Tokens(로고/컬러 적용: 헤더/배지/CTA에 제한적으로)
- 참여 진행 바(progress bar): 전체 기간 대비 진행률
- 프로그램 일정 타임라인(오늘/이번 주)
- 역할 관계 상태 카드:
  - 담당 코디네이터 정보(사진/이름/연락/근무시간)
  - 보호자 연결 상태(연결됨/미연결/승인대기)
- 티켓/문의/댓글 모더레이션 컴포넌트(상태/담당/우선순위)
- 체크리포트 제출 워크플로우(초안/제출/반려/완료)
- 예약/결제 상태 배지(단계별)
- 데이터 분리 안내(사용자 혼란 방지 문구):
  - “현재 고라데이 캠프 기준 정보입니다” 같은 중립 안내(작게)

========================================================
8) 화면 산출 요구(반드시 생성해야 할 화면 목록)
========================================================

--------------------------------------------------------
A) 공통(인증/역할/모드 진입)
--------------------------------------------------------
S1. 로그인/가입(공통)
- 역할 선택이 아니라 “계정” 로그인 후 시스템이 역할 권한을 부여(관리자/코디는 별도 초대)
- 일반 사용자는 참가자/보호자 플로우로

S2. 모드 자동 분기 진입 화면
- 로딩 후:
  - 활성 참여 0개: NORMAL_MODE 홈 진입
  - 활성 참여 1개: “입장(Enter Village)” 트랜지션 후 해당 VILLAGE_MODE 홈 진입
  - 활성 참여 2개 이상: “현재 참여 공간 선택” 화면(활성 참여 카드 리스트)

S3. 현재 참여 공간 선택(활성 참여 다중일 때)
- 활성 참여 카드: 마을 로고/마을명/프로그램명/기간/남은일수
- ‘입장’ 버튼
- 하단: “예정/과거 참여 일정 보기”

--------------------------------------------------------
B) NORMAL_MODE — 참가자 중심(기존 기능 유지 + 참여/예약 흐름 추가)
--------------------------------------------------------
N1. 일반 홈(개인 대시보드) [기존 유지]
- 내 컨디션/수면/스트레스/자율신경/피로/생체리듬(성인 톤)
- 상단에 “현재 참여 마을 없음” 또는 “다음 참여 예정” 카드(있다면)

N2. 캠프/프로그램 찾기(탐색 홈)
- 추천 마을/추천 프로그램
- 필터(지역/기간/목적: 수면/리듬/스트레스/인지/정서/사회활동)
- 마을/프로그램 카드를 명확히 분리(마을 소개 vs 프로그램 상품)

N3. 마을 상세(소개)
- 마을 로고/브랜드 컬러
- 시설/특징/후기
- 운영 프로그램 리스트(기간형 상품)

N4. 프로그램 상세(예약/결제)
- 기간(start/end), 가격, 포함(식단/숙소/프로그램)
- 준비물/유의사항/취소/환불 정책
- 예약 버튼 → 결제 플로우

N5. 예약/결제 플로우
- 예약 정보 확인 → 결제 수단 → 결제 완료
- 결제 완료 화면:
  - “참여 확정(Participation 생성됨)” + 시작일까지 D-n
  - “내 참여 일정 보기” CTA

N6. 내 예약/결제
- 상태별 탭: 결제대기/결제완료/취소/환불
- 각 항목에서 “참여 확정/대기/진행/완료” 상태 표시

N7. 내 참여 이력(타임라인)
- 과거/현재/미래 일정
- 현재 참여 강조, 예정 참여는 시작 D-n 표시
- 각 참여 카드에서 “입장” 또는 “상세 보기”

N8. 가족/연결(참가자-보호자 연결 관리) [공통이나 NORMAL_MODE에 노출]
- 보호자(자녀) 초대 코드/QR 생성(유효기간)
- 연결 요청 승인/거절
- 공유 동의(Consent) 설정:
  - 일반 모드 공유 범위(요약/위험만/비공유)
  - 마을 참가 모드 공유 범위(요약/위험만/캠프별 예외)
  - 기간 종료 후 접근 정책(기본: 접근 종료, 옵션: 요약만 유지)

--------------------------------------------------------
C) VILLAGE_MODE — 참가자(40+) 마을 전용 대시보드(요구사항 4 반영)
--------------------------------------------------------
V-P1. 마을 전용 대시보드(참가자)
- 상단 Context Bar(필수)
- 마을 로고 + 브랜드 컬러 적용(헤더/배지 수준)
- 참여 진행 바(progress bar): 오늘 포함 진행률
- “남은 일정” 텍스트(D-n)
- 오늘 프로그램 일정 타임라인(오늘 세션 3개 정도 요약)
- 담당 코디네이터 카드(연락/운영시간/문의)
- 보호자 연결 상태 카드(연결됨/미연결/승인대기)
- 공지/문의/댓글 섹션 활성화(마을 컨텍스트)
- 하단: “내 컨디션(오늘 요약)” 카드(기존 컴포넌트 재사용)

V-P2. 프로그램 일정(참가자)
- 오늘/이번 주 타임라인
- 세션 상세(장소/준비물)
- (옵션) 체크인/참여 표시

V-P3. 식단(참가자 보기)
- 오늘/이번 주 식단 카드(아침/점심/저녁)
- 변경 공지 표시

V-P4. 숙소(참가자 보기)
- 배정 정보(동/호/규칙)
- 안내 공지/문의 링크

V-P5. 공지(참가자)
- 공지 리스트/상세
- 댓글(허용 시)

V-P6. 문의/건의(참가자)
- 문의 작성
- 내 문의 상태(대기/답변/완료)
- 민감정보 안내 문구

V-P7. 참여 종료(수료/완료) 화면
- 기간 종료 시 자동 노출
- 요약 리포트(이번 참여 동안의 변화 요약)
- 감사/다음 참여 추천(부드럽게)
- “일반 모드로 돌아가기” CTA
- 자연스러운 전환 애니메이션

--------------------------------------------------------
D) 보호자(자녀) — 기간 종속 접근 + 마을 컨텍스트 분리(요구사항 3 반영)
--------------------------------------------------------
G1. 보호자 홈(연결된 참가자 목록)
- 참가자 카드 + “현재 참여중 마을/프로그램” 배지(있을 때만)
- 접근 가능 기간 표시(예: 03.10~03.17)

G2. 참가자 선택 후 “보기 기준 선택”
- 일반 보기(동의 범위 내)
- 참여중 마을 보기(활성 참여 목록)
- 예정/과거 참여 보기(정책에 따라 제한)

G3. 마을 참여 요약 대시보드(보호자)
- 상단 Context Bar(필수)
- 오늘 상태(정상/주의/관리필요) + 최근 7일 변화(동의 범위 내)
- 참여 진행 바 + 오늘 일정 요약(불안 유발 없이 정보형)
- 코디네이터 문의/예약 CTA
- 보호자 ↔ 코디네이터 메시지/요청 플로우

G4. 접근 제한/기간 종료 화면(보호자)
- 참여 기간 종료 시:
  - “접근 기간이 종료되었습니다”
  - (옵션) 참가자가 허용한 경우 ‘요약 리포트만 보기’ 제공
  - 그렇지 않으면 종료 안내만 제공

G5. 보호자 문의/요청(마을 컨텍스트)
- 문의 유형(건강/일정/숙소/기타)
- 상태 추적(티켓)

--------------------------------------------------------
E) 코디네이터 — 캠프(프로그램) 운영 OS (참가자 화면 복제 금지)
--------------------------------------------------------
코디네이터는 “자기 마을 1개(고정) + 배정된 프로그램(캠프) 운영”이 핵심.
코디네이터는 마을/프로그램 컨텍스트가 없으면 운영 기능이 노출되면 안 됨.

C0. 코디네이터 홈(배정 프로그램 목록)
- 상단: 내 마을 고정 헤더(전환 없음)
- 배정 프로그램 리스트(진행중/예정/종료)
- 프로그램 선택 → VILLAGE_MODE 운영으로 입장(Enter Village 트랜지션 사용 가능)

C1. 캠프보드(코디네이터 운영 대시보드)
- Context Bar 필수
- 오늘 일정 타임라인(세션 상태: 예정/진행/완료/지연)
- KPI: 참가자 수, 출석률, 측정 완료율, 주의/관리필요 수
- 오늘 우선 확인 대상(주의/관리필요/측정누락/불참)
- 보호자 요청/문의 큐

C2. 체크리포트 허브(오전/저녁/일일 요약)
- 제출 상태(초안/제출/반려/완료)
- 관리자에게 제출 플로우(Closed Loop)
- AI 요약 UI(과장 금지) + 코디 편집 가능

C3. 오전체크 리포트 작성(참가자 리스트 기반)
- 빠른 입력(3단 선택/토글/메모)
- 제출 버튼

C4. 저녁체크 리포트 작성
C5. 일일 요약 리포트(자동요약 + 편집 + 제출)

C6. 세션 진행(체크인/불참 사유/특이사항)
- 세션별 체크인
- 종료 후 세션 리포트 자동 생성(참여율/이슈)

C7. 참가자 진행 상황(운영용 리스트)
- 숙소/조/체크 상태/위험 배지
- 상세에서 “현장 기록/조치/관리자 에스컬레이션” 가능

C8. 보호자와의 커뮤니케이션(권한 기반)
- 보호자 문의 리스트
- 코디 답변 템플릿(정중, 진단 표현 금지)
- 필요 시 관리자에게 전달

C9. 이슈/인시던트 관리(티켓)
- 상태(대기/처리중/완료)
- 담당/기록/리포트 포함

--------------------------------------------------------
F) 관리자(가맹주) — “내 마을 1개만” 운영 가능한 콘솔(요구사항 5 완전 반영)
--------------------------------------------------------
관리자는 마을 전환 UI 금지. 항상 “내 마을 고정”.
마을 내 프로그램(캠프)별 운영을 관리한다.

A0. 관리자 홈(내 마을 고정)
- 로고/마을명 고정 헤더(전환 없음)
- 진행중 프로그램 카드 리스트
- 문의/댓글/결제/데이터 수집 현황 요약

A1. 운영 대시보드(마을/프로그램 관점)
- 마을 전체 KPI: 진행중 프로그램 수, 예약/결제 현황, 문의 처리율, 데이터 수집 통계
- 프로그램별 KPI 카드(참가자 수/출석/리포트 제출/이슈 수)
- 프로그램 선택 시 프로그램 컨텍스트 운영 화면으로 진입(Context Bar 적용)

A2. 프로그램(캠프) 관리 목록(CRUD)
- 생성/수정/삭제
- 프로그램 기간(start/end), 정원, 가격, 상태(예정/진행/완료)
- 코디네이터 배정

A3. 프로그램 상세/설정
- 소개/규정/환불/준비물
- 체크리포트 제출 규칙(오전/저녁/일일)
- 공지 기본 템플릿

A4. 프로그램 일정(세션) 관리(캘린더/리스트)
- 세션 생성/수정/삭제
- 담당 코디/장소/정원
- 변경 시 공지 발행 옵션

A5. 식단 관리(주간 캘린더 CRUD)
- 아침/점심/저녁 메뉴 추가/수정/삭제/복제
- 변경 공지 자동 생성
- 참가자 공지 표시

A6. 숙소 관리(구조 CRUD)
- 동/층/호/침대 생성/수정/삭제
- 수용 인원

A7. 숙소 배정(그리드 + 이동)
- 참가자 배정/이동/퇴실
- 점유율/빈자리/과밀 경고
- 변경 로그

A8. 참가자 목록(프로그램별)
- 참가자 상태(참여확정/대기/진행/완료)
- 측정 완료율/이슈 여부
- 보호자 연결 여부 표시

A9. 보호자 목록(프로그램별)
- 연결 상태/접근 기간
- 문의 이력

A10. 코디네이터 관리
- 코디 계정 생성/비활성화
- 프로그램 배정
- 권한(공지 작성 가능/문의 처리 가능 등)

A11. 예약 및 결제 상태 확인(요구사항 포함)
- 예약/결제 리스트(결제대기/결제완료/취소/환불)
- 참여 확정 자동 생성 여부 표시
- 환불 처리 상태(표시용)

A12. 데이터 수집 통계(요구사항 포함)
- 측정/워치 동기화 통계(프로그램별)
- “오늘 수집률/누락률” 카드
- 누락자 리스트(코디에게 공유 버튼)

A13. 문의/건의 관리(티켓)
- 접수/담당/답변/완료
- 보호자/참가자 구분
- 내부 메모

A14. 댓글 관리(모더레이션)
- 공지/문의/게시판 댓글 스트림
- 신고/숨김/삭제/차단 + 사유 기록
- 모더레이션 로그

A15. 공지사항 관리(CRUD)
- 작성/수정/삭제/예약
- 대상(참가자/보호자/코디)
- 댓글 허용 토글/상단 고정

A16. 체크리포트 검토/반려/완료(Closed Loop)
- 코디 제출 리포트 목록
- 반려 사유/코멘트
- 완료 처리 후 대시보드 반영

A17. 리포트 센터(PDF/공유)
- 프로그램 운영 리포트
- 문의 처리 리포트
- 참여/결제 요약 리포트

--------------------------------------------------------
G) (옵션) PlatformAdmin 백오피스 — 멀티 마을 확장 필수(요구사항 7 보강)
--------------------------------------------------------
이 역할은 일반 앱 사용자에게 노출 최소화(별도 페이지).
PA1. 치유마을 생성/관리(100+ 확장)
PA2. 마을 관리자(가맹주) 계정 생성 및 village_id 할당(핵심)
PA3. 데이터 테넌트 분리 정책 표시(안내)
PA4. 마을 브랜딩 토큰 관리(로고/컬러)

========================================================
9) 상태/엣지케이스(반드시 UI로 표현)
========================================================
- 활성 참여 없음: NORMAL_MODE, 참여 유도 카드
- 결제 완료했으나 시작 전: “참여 확정/대기” 화면 + 시작 D-n
- 활성 참여 2개 이상: “현재 참여 공간 선택” 강제
- 참여 종료: 보호자 접근 종료 + 수료 화면 + 모드 전환
- 권한 없음:
  - 관리자가 다른 마을 데이터 접근 불가(메뉴 자체가 없음)
  - 보호자가 기간 외 접근 시 종료 안내
- 데이터 없음/동기화 지연:
  - “최근 수집 없음(마지막 수집 시간)” 표시 + 안내
- 댓글/문의 폭주:
  - 상태 필터/우선순위/담당자 배정 UI

========================================================
10) 프로토타입 연결(중요 플로우)
========================================================
F1. NORMAL_MODE → 탐색 → 프로그램 상세 → 예약/결제 → 참여 확정(대기) → 시작일에 VILLAGE_MODE 자동 진입
F2. VILLAGE_MODE 진입(입장 트랜지션) → 마을 대시보드 → 일정/공지/문의
F3. 참여 종료 → 수료 화면 → NORMAL_MODE 복귀
F4. 참가자-보호자 연결: 초대코드 생성 → 보호자 입력 → 참가자 승인 → 기간/캠프 기반 접근
F5. 코디 체크리포트 제출 → 관리자 검토/반려/완료(Closed Loop)
F6. 관리자 CRUD: 프로그램 생성/기간 설정 → 일정 편성 → 식단/숙소 배정 → 공지 발행 → 문의/댓글 처리

========================================================
11) 피그마 파일 구조(페이지 구성)
========================================================
Page 1: Design System + Theme Tokens(마을 브랜딩 적용 구조)
Page 2: NORMAL_MODE (탐색/예약/결제/내참여)
Page 3: VILLAGE_MODE Participant (마을 대시보드/일정/식단/숙소/공지/문의/종료)
Page 4: Guardian (기간 종속 접근 + 캠프 컨텍스트)
Page 5: Coordinator OS (캠프 운영/체크리포트/이슈/문의)
Page 6: Admin Console (마을 고정 + CRUD/배정/모더레이션/결제/통계)
Page 7: (Optional) PlatformAdmin Backoffice
Page 8: User Flows & States (모드 전환/기간/권한/결제→참여)

최종 출력:
- 고해상도 하이파이 UI
- 주요 흐름 프로토타입 연결
- “모드/기간/마을 컨텍스트/권한 분리”가 화면만 봐도 즉시 이해되는 수준으로 설계할 것.

이 요구사항을 기반으로 전체 UI를 생성하라.
